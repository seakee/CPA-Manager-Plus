## Overview

This release focuses on reliable configuration updates and accurate usage identification. Consecutive OAuth model-alias and AI Provider operations no longer overwrite one another, Codex usage is aggregated from the requested tier, and monitoring distinguishes multi-key channel identities more clearly.

## Highlights

### Features

- Add keyboard-accessible, viewport-aware information popovers to quota cards. Plan, reset-credit, and expiry information now remains more compact and readable on narrow screens (`web/quota`).

### Fixes

- Serialize OAuth model-alias updates and align validation with CPA rules. Multi-channel edits make a best-effort rollback after failed or ambiguous results, preventing saved configuration from being lost during consecutive edits (`web/auth-files`).
- Coordinate AI Provider enablement, priority, and health-check updates through a shared lock so overlapping optimistic writes cannot overwrite newer Provider configuration (`web/ai-providers`).
- When Codex request and response tiers differ, Usage Analytics and Manager Server aggregate usage from the requested tier. Other Provider types retain response-tier precedence (`manager-server/usage`, `web/usage`).
- Make Request Monitoring and Account Overview prefer disambiguated multi-key source identities, including OAuth email identities for Codex and xAI, so channels no longer collapse into generic labels (`web/monitoring`).

### CI

- Send the reviewed repository Telegram release post only after both native-package and Docker-image jobs succeed. Missing configuration or delivery failures do not roll back an already-created Release (`ci/release`).
- Make PR checks select relevant jobs by changed paths and cover workflow, native-packaging-script, and Docker Compose changes to avoid missing infrastructure validation (`ci/pr-check`).

## Upgrade Notes

- This release introduces no SQLite schema, persistent-format, `data.key`, or required configuration migration.
- For new Codex usage records where requested and response tiers differ, aggregation uses the requested tier. Existing stored records are not automatically recalculated.
